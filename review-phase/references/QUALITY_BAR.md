# Quality Bar for the Review Phase

A good review is:

- evidence-based
- source-of-truth driven
- independent from implementation enthusiasm
- explicit about business fulfillment, not only technical correctness
- clear about severity and next action
- concise enough to scan quickly
- specific enough that follow-up work is obvious

A weak review typically has one or more of these failures:

- starts from code impression instead of intent
- checks only code quality or only tests
- ignores business rules, user workflow, or success criteria
- does not compare implementation against scope and non-goals
- lists issues without severity
- gives no acceptance decision
- produces findings that are hard to act on

## Acceptance bar

A review is good enough only if it can answer all of these:

1. Was the right thing built?
2. Was it built correctly enough?
3. Did it stay aligned with the approved plan / roadmap / design?
4. Are the remaining gaps clear and prioritized?
5. Is the audience told what to do next?
